# 企业AI智造平台 — 方案设计文档

## 一、项目概述

### 1.1 平台定位

**企业AI智造平台** 是一站式企业级 AI 应用平台，通过统一的前端门户，将三大开源引擎（OpenCode、DataEase、BuildingAI）与平台自身整合为有机整体，覆盖"代码生成 → 数据分析 → 智能决策"的完整链路。

- **"智造"** = 智能 + 制造/创造：用 AI 能力赋能企业数字化创造
- **目标用户**：企业开发团队、数据分析师、业务决策者、AI 应用搭建者
- **核心价值**：一个平台完成编码、分析、智能体搭建，消除工具切换成本
- **当前实现状态**：代码仓库中已实现了 OpenCode 多用户网关 + Keycloak 认证；统一前端、DataEase/BuildingAI 集成、智能流水线尚未实现

**新增开发量**：
1. BFF 网关层 — 在现有 OpenCode 网关（Bun + Hono，端口 `9090`）基础上扩展为 tri-system gateway，增加 DataEase/BuildingAI 代理与 token 交换；目标端口 `3001`
2. 统一前端 SPA — 全新设计的交互界面（当前尚未实现），目标端口 `3000`
3. Keycloak 统一认证中心 — 从现有 `opencode` realm（端口 `9091`）演进为 `aiplatform` realm（端口 `8080`），覆盖三个子系统
4. 后端零改动，仅作为反向代理目标（OpenCode 无认证；DataEase/BuildingAI 权限由各自系统 + Keycloak 共同管控）
5. 平台数据库 — MySQL 8.0，存储 token 映射、审计日志、流水线、API key、通知配置等

> 端口说明：当前代码中 gateway 运行在 `9090`、Keycloak 运行在 `9091`。目标架构统一为 `3000`（前端）/`3001`（gateway）/`8080`（Keycloak）/`:4096`（OpenCode 单实例）。演进时先将当前 per-user OpenCode 进程简化为单实例 :4096。

## 二、菜单结构与功能定义

> 本节基于四个子系统代码现状重新设计：
> - **OpenCode**：AI coding agent（session + tools + web PTY + diff review），无用户/权限/认证
> - **DataEase**：BI 平台（dashboard/chart/dataset/datasource/report + RBAC），使用 `X-DE-TOKEN` JWT
> - **BuildingAI**：AI agent 平台（agent/knowledge base/model/MCP + console RBAC），使用自定义 JWT
> - **平台本 app**：当前已提供 OpenCode 多用户网关 + Keycloak，待扩展为 tri-system gateway + 统一前端

### 2.2 各模块详细功能

> 以下对每个二级菜单进行详细功能设计，包含：功能目标、用户流程、页面/组件、数据来源、权限控制、异常处理。

#### 2.2.1 工作台

##### 关键指标卡片
- **功能目标**：让用户进入平台后第一时间看到三系统的核心运行状态
- **用户流程**：用户登录 → 进入工作台 → 看到 4 张指标卡片
- **页面组件**：4 张大卡片（活跃会话数、仪表板数量、智能体数量、流水线运行次数）
- **数据来源**：OpenCode `GET /api/session`、DataEase `/de2api/panel`、BuildingAI `/api/agent`、平台 MySQL 流水线表
- **权限控制**：所有登录用户可见，但卡片内容根据平台角色过滤（如业务用户看不到代码工场会话数）
- **异常处理**：某个子系统不可用时，对应卡片显示"—"并标红

##### 三模块快捷入口
- **功能目标**：一键跳转代码工场 / 数据洞察 / AI大脑
- **用户流程**：点击卡片 → 跳转到对应模块首页
- **页面组件**：3 张带渐变色的大卡片
- **数据来源**：前端路由配置
- **权限控制**：根据平台角色显示可用入口（开发者看到代码工场，数据分析师看到数据洞察和 AI大脑）
- **异常处理**：无

##### 最近活动时间线
- **功能目标**：展示用户在三个系统中的最近操作记录
- **用户流程**：用户登录 → 滚动查看时间线
- **页面组件**：垂直时间线，每条记录包含时间、系统、操作、结果
- **数据来源**：BFF 审计日志（MySQL `audit_log` 表）
- **权限控制**：只展示当前用户有权限查看的系统的操作记录
- **异常处理**：无

##### 服务健康状态监控
- **功能目标**：实时显示三个后端服务是否在线
- **用户流程**：用户登录 → 顶部三色指示灯显示状态
- **页面组件**：顶部指示灯 + 悬浮详情面板
- **数据来源**：BFF `/api/gateway/health` 每 30 秒探测一次
- **权限控制**：所有登录用户可见
- **异常处理**：服务不可用时变红，点击显示最后探测时间和错误信息

##### 全局搜索 / 命令面板
- **功能目标**：通过 Cmd/Ctrl+K 快速搜索功能、会话、仪表板、智能体
- **用户流程**：按快捷键 → 输入关键词 → 选择结果 → 跳转
- **页面组件**：居中浮层搜索框，结果分组显示
- **数据来源**：前端路由表 + 三系统 API 索引（可选）
- **权限控制**：只返回用户有权限访问的结果
- **异常处理**：三系统搜索失败时，仅显示前端本地功能

#### 2.2.2 代码工场

##### AI 编程会话
- **功能目标**：按用户已分配项目进入 OpenCode 编程会话；会话 ID 落库本平台，便于按用户/项目过滤
- **用户流程**：
  1. 进入代码工场 → AI 编程会话 → 展示当前用户已分配的活跃项目卡片
  2. 点击项目卡片 → `POST /api/projects/:id/open-chat`：优先打开本库已绑定会话；否则取该项目目录下最早 OpenCode 会话；没有则新建并写入 `opencode_session`
  3. 以 iframe 嵌入 OpenCode 会话页（裁切顶栏 DEV/标签条；可选裁切右侧文件变更区）
  4. 顶栏「会话列表」展开左侧「我的会话」：切换会话、新建会话、删除会话（本平台软删）；「切换项目」回到项目卡片
  5. 在嵌入会话内与 agent 对话、查看变更（OpenCode 原生能力）
  6. 支持从「会话列表」深链进入：`/code-factory/chat?projectId=&sessionId=`
- **页面组件**：项目卡片网格、会话顶栏、左侧「我的会话」侧栏、iframe 嵌入区（含文件变更显隐控件）
- **数据来源**：平台 MySQL `project` / `project_member` / `opencode_session`；OpenCode `GET/POST /session`；嵌入 URL `/server/{base64(origin)}/session/{sessionId}`
- **权限控制**：仅项目成员可打开；会话映射按 `platform_user_id` + `project_id` 隔离
- **异常处理**：OpenCode 不可用时提示打开失败；无分配项目时引导去「项目初始化」

##### 会话列表/历史
- **功能目标**：跨项目总览当前用户的活跃会话映射（找历史 → 打开进入 AI 编程会话）；与「AI 编程会话」侧栏互补——侧栏管当前项目即时切换，本页管跨项目检索
- **用户流程**：
  1. 进入代码工场 → 会话列表 → `GET /api/sessions` 加载「我的」全部活跃映射
  2. 左侧按项目筛选；顶部按标题/sessionId 搜索
  3. 点击「打开」→ `/code-factory/chat?projectId=&sessionId=` 深链进入嵌入对话
  4. 删除确认后调用 `DELETE /api/projects/:id/sessions/:sessionId`（本平台软删）
- **页面组件**：项目筛选侧栏、搜索框、会话表格（标题/项目/更新时间/打开/删除）、空态引导
- **数据来源**：平台 `GET /api/sessions`（`opencode_session` join `project`）；删除复用项目级 DELETE；打开走 chat 深链
- **权限控制**：仅返回当前用户自己的 `status=active` 映射；不能看到他人会话
- **异常处理**：OpenCode/网关异常时显示错误；空列表引导去「AI 编程会话」或「项目初始化」；深链会话缺失时回退 `open-chat`

##### 代码审查
- **功能目标**：在平台内对已分配项目发起 OpenCode `review`，审查工作区 diff / 指定分支或 PR
- **用户流程**：进入代码工场 → 代码审查 → 选项目（可选本人会话）→ 可选填审查参数（空=未提交变更；可填分支/PR）→「开始审查」→ 查看结果文本与风险标签
- **页面组件**：项目/会话选择器、参数输入、风险等级芯片、审查结果正文
- **数据来源**：平台 `POST /api/projects/:id/review`（Gateway 校验项目权限后调用 OpenCode `POST /session/{id}/command`，`command=review`；空变更时先用 `GET /vcs/diff?mode=git` 短路）
- **权限控制**：菜单随代码工场；API 仅当前用户可访问的项目；会话须为本人映射
- **异常处理**：无 git 变更时提示「没有可审查的 diff」；OpenCode 不可达/命令失败时展示错误文案

##### 变更 Diff
- **功能目标**：查看本人会话中 agent 产生的文件差异（非任意 git 工作区浏览）
- **用户流程**：进入代码工场 → 变更 Diff → 选项目与本人会话 → 左侧文件列表 → 右侧 patch 高亮
- **页面组件**：项目/会话选择器、文件列表（+/- 统计与 status）、patch 视图
- **数据来源**：平台 `GET /api/projects/:id/sessions/:sessionId/diff` → OpenCode `GET /session/{id}/diff`
- **权限控制**：仅本人 `opencode_session` 活跃映射；他人会话返回 404
- **异常处理**：无变更时显示空状态；会话不存在/无权限时提示重新选择

##### 项目初始化
- **功能目标**：管理平台级项目工作空间——新建项目、绑定服务器绝对路径、多用户共享并分配角色；数据存本平台 MySQL。AGENTS.md / OpenCode `/init` 生成本阶段不做
- **用户流程**：进入代码工场 → 项目初始化 → 单步创建（名称、绝对路径下拉、可选模板、分配成员）→ 编辑弹窗维护信息与成员 → 保存成功提示并关闭
- **页面组件**：项目列表（搜索/状态过滤）+ 创建/编辑弹窗（信息 + 成员；候选用户来自 Keycloak 同步）
- **数据来源**：平台 MySQL 表 `project` / `project_member` / `project_template`；API `GET/POST /api/projects`、`/project-paths`、`/project-candidate-users` 等
- **权限控制**：菜单仅超级管理员和开发者可见；项目级角色为所有者/管理员/成员/观察者（owner/admin/member/viewer）；非超管仅能看到自己加入的项目
- **异常处理**：路径非绝对路径、目录不存在或不可访问、路径已被占用时拒绝创建并提示

##### GitHub 自动化
- **功能目标**：提供 OpenCode GitHub Action / PR 自动化的安装向导（生成工作流 YAML），由用户提交到目标仓库；本平台不代跑 `opencode github` CLI、不托管 GitHub OAuth
- **用户流程**：进入代码工场 → GitHub 自动化 → 勾选触发条件 → 预览/复制/下载 `.github/workflows/opencode.yml` → 按步骤配置 Secrets 与权限并提交到仓库
- **页面组件**：触发条件勾选、YAML 预览、复制/下载按钮、安装步骤说明
- **数据来源**：平台模板生成（`GET /api/code-factory/github-workflow` 或前端同逻辑）；Action 使用官方 `sst/opencode/github`
- **权限控制**：随代码工场菜单（`code_factory.github`）
- **异常处理**：无远端安装失败态（安装在用户仓库侧完成）；复制失败时提示手动选中文本

##### 配置管理
- **功能目标**：在平台内嵌入 OpenCode 工作台/配置界面
- **用户流程**：进入代码工场 → 配置管理 → 顶栏显示「配置管理」，主区全屏 iframe 加载 `http://127.0.0.1:4096/`（无页面内标题/描述；裁切 OpenCode DEV 顶栏）
- **页面组件**：顶栏页标题 + 全屏 iframe（平台侧裁切约 36px 顶栏）
- **数据来源**：OpenCode Web UI（直连本地 4096）；不修改 OpenCode 源码
- **权限控制**：随代码工场菜单权限（`code_factory.config`）
- **异常处理**：OpenCode 未启动时 iframe 空白/无法加载，需保证本地服务可用

##### 使用统计
- **功能目标**：展示代码工场的使用趋势
- **用户流程**：进入代码工场 → 使用统计 → 查看 7 天/30 天趋势图
- **页面组件**：折线图 + 指标卡片
- **数据来源**：平台 MySQL 审计日志 + OpenCode API
- **权限控制**：仅超级管理员和开发者可见
- **异常处理**：无

> **不应承诺的功能**（OpenCode 当前不具备）：项目模板市场、代码片段库、Copilot 式自动补全、企业 RBAC、审批流。

#### 2.2.3 数据洞察

##### 仪表板中心
- **功能目标**：集中管理所有 BI 仪表板
- **用户流程**：进入数据洞察 → 仪表板中心 → 查看/搜索/创建/编辑仪表板
- **页面组件**：卡片网格 + 搜索框 + 新建按钮
- **数据来源**：DataEase `/de2api/panel`
- **权限控制**：根据 DataEase 内部 RBAC 过滤
- **异常处理**：DataEase 不可用时显示"服务不可用"

##### 图表库
- **功能目标**：浏览和编辑已创建的图表
- **用户流程**：进入数据洞察 → 图表库 → 点击图表编辑
- **页面组件**：图表列表 + 预览 + 编辑器
- **数据来源**：DataEase `/de2api/chart`
- **权限控制**：根据 DataEase 内部 RBAC 过滤
- **异常处理**：无

##### 数据集
- **功能目标**：管理数据集，包括创建、字段配置、权限

## 三、架构设计

### 3.1 API 网关路由规划

```
/api/gateway/health          → 聚合三个后端健康状态
/api/gateway/stats           → 聚合统计信息
/api/gateway/auth/me         → 当前用户信息
/api/gateway/auth/roles      → 当前用户角色

/api/code/*                  → 反向代理 → OpenCode :4096（无认证，直接转发）
/api/bi/*                    → 反向代理 → DataEase :8100（注入 X-DE-TOKEN）
/api/agent/*                 → 反向代理 → BuildingAI :4091（注入 Authorization: Bearer）

/auth/*                      → Keycloak 认证服务 :8080
```

### 3.2 跨系统数据流

```
┌──────────────────────────────────────────────────────────┐
│                    智能流水线引擎                          │
│                                                          │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐              │
│  │ 触发器  │───▶│ 编排器  │───▶│ 执行器  │              │
│  │(定时/   │    │(DAG工作 │    │(按序调用│              │
│  │ 事件/   │    │ 流定义) │    │ 各API)  │              │
│  │ 手动)   │    └─────────┘    └────┬────┘              │
│  └─────────┘                        │                   │
│                         ┌───────────┼───────────┐       │
│                         ▼           ▼           ▼       │
│                    ┌────────┐ ┌────────┐ ┌────────┐     │
│                    │OpenCode│ │DataEase│ │BuildAI │     │
│                    │  API   │ │  API   │ │  API   │     │
│                    └────────┘ └────────┘ └────────┘     │
│                                                          │
│  步骤间数据通过上下文对象传递:                              │
│  ctx = {                                                 │
│    step1_output: DataEase查询结果,                        │
│    step2_output: BuildingAI分析结果,                      │
│    step3_output: OpenCode生成的报告代码                    │
│  }                                                       │
└──────────────────────────────────────────────────────────┘
```

**流水线实现要点**：
- 持久化：使用数据库保存流水线定义、执行状态、任务队列
- 序列化：步骤间数据以 JSON 传递；大数据集使用对象存储或临时文件
- 错误处理：某一步失败时停止、重试或跳过，并通知触发者
- 沙箱：执行 OpenCode 生成的代码时，必须在隔离环境（容器/沙箱）中运行
- 触发器：支持定时（cron）、Webhook、事件驱动、手动触发

```yaml
认证策略:
  方式: Keycloak 作为统一身份提供者（IdP）
  协议: OpenID Connect / OAuth2
  流程:
    - 用户通过统一前端登录，Keycloak 签发 JWT
    - BFF 验证 JWT，提取用户身份、角色、组织
    - BFF 将身份信息通过 header 或 token 转发给下游系统

  各系统认证策略:
    - OpenCode: 本身无认证、无权限管理，BFF 直接反向代理，无需身份转换
    - DataEase: 使用 X-DE-TOKEN JWT 认证；通过 BFF 作为 OIDC 代理终端换取 DataEase token
    - BuildingAI: 使用自定义 JWT + DB token registry 认证；通过 BFF 做 JIT 用户预创建后获取 token

  下游系统集成:
    - OpenCode: BFF 直接代理 /api/code/*，无认证；平台前端仅做菜单级准入控制
    - DataEase:
        1. BFF 作为 OIDC 代理终端，用 Keycloak 完成用户认证
        2. BFF 将 Keycloak 用户信息放入 `X-Userinfo` header
        3. BFF 调用 `POST /de2api/login/platformLogin/2` 换取 DataEase `X-DE-TOKEN`
        4. BFF 缓存 `X-DE-TOKEN`，并在后续代理 `/api/bi/*` 时注入该 header
        5. 启用 DataEase `basic.autoCreateUser`，配置 `platformOid` / `platformRid`
       注意：完整 OIDC 支持需要 DataEase Enterprise / XPack 版本；社区版仅支持 admin substitute 登录
    - BuildingAI:
        1. BFF 用 Keycloak 验证用户身份
        2. BFF 根据 email/username 在 BuildingAI 中 JIT 创建或查找用户
        3. BFF 调用 BuildingAI 登录逻辑生成 BuildingAI JWT（写入 `user_token` 表）
        4. BFF 缓存 BuildingAI JWT，并在后续代理 `/api/agent/*` 时注入 `Authorization: Bearer <token>`
        5. 将 Keycloak role 映射为 BuildingAI 默认 role

  用户角色:
    - 超级管理员: 全部权限
    - 开发者: 代码工场 + 系统管理(只读)
    - 数据分析师: 数据洞察 + AI大脑
    - 业务用户: AI大脑 + 智能流水线(只执行)

  角色映射:
    - Keycloak realm roles ↔ 平台角色
    - Keycloak groups ↔ 组织/部门
    - DataEase: Keycloak 角色/组映射到 `platformRid`（默认角色）和 `platformOid`（默认组织）
    - BuildingAI: Keycloak 角色映射到 BuildingAI 的单一 role；dataset/agent/MCP 权限在 BuildingAI 内部管理

  审计:
    - Keycloak 记录登录/登出事件
    - BFF 记录所有 API 调用：用户、时间、下游系统、接口、操作结果
    - DataEase / BuildingAI 本地审计日志保留原有记录
```

### 3.4 权限控制矩阵与集成方式

#### 3.4.1 平台角色权限矩阵

| 平台角色 | 代码工场 (OpenCode) | 数据洞察 (DataEase) | AI大脑 (BuildingAI) | 智能流水线 | 系统管理 |
|----------|---------------------|---------------------|---------------------|------------|----------|
| 超级管理员 | 全部权限 | 全部权限 | 全部权限 | 全部权限 | 全部权限 |
| 开发者 | 全部权限 | 无权限 | 无权限 | 查看 | 只读 |
| 数据分析师 | 无权限 | 全部权限 | 全部权限 | 查看/执行 | 无权限 |
| 业务用户 | 无权限 | 查看 | 查看 | 只执行 | 无权限 |

> 说明：
> - OpenCode 本身无认证、无权限体系，平台仅控制"能否看到代码工场菜单"；进入后拥有 OpenCode 全部能力
> - DataEase 权限分两层：平台级控制（能否进入）+ DataEase 内部 RBAC（组织、角色、数据权限、资源权限）
> - BuildingAI 权限分两层：平台级控制（能否进入）+ BuildingAI 内部 RBAC（角色、应用、知识库、模型、MCP 权限）

#### 3.4.2 OpenCode 权限控制

- OpenCode 本身无认证、无用户、无角色、无权限管理
- 目标架构采用单实例 OpenCode 运行在 `:4096`，BFF 直接反向代理
- 平台前端通过平台角色控制"代码工场"菜单是否显示；允许进入即拥有 OpenCode 全部能力
- BFF 在 `/api/code/*` 前做平台身份校验，避免未授权访问
- 操作审计（谁、何时、调用了什么 API）由 BFF 统一记录
- 当前代码中 OpenCode 是每用户一个独立进程（动态端口）；演进时需简化为单实例 :4096

#### 3.4.3 DataEase 权限集成

- DataEase 使用 `X-DE-TOKEN` JWT 进行 API 认证，payload 包含 `uid`（用户ID）和 `oid`（默认组织ID）
- BFF 作为 OIDC 代理终端，通过 `POST /de2api/login/platformLogin/2` 换取 `X-DE-TOKEN`
- 首次登录时通过 `basic.autoCreateUser` 自动创建用户，并分配 `platformOid`（默认组织）和 `platformRid`（默认角色）
- Keycloak 的 group 可映射到 DataEase 的"组织"
- Keycloak 的 role 可映射到 DataEase 的"角色"
- 数据级、资源级权限（仪表板、数据集、数据源）仍在 DataEase 内部管理
- 注意：完整 OIDC 支持需要 DataEase Enterprise / XPack 版本

#### 3.4.4 BuildingAI 权限集成

- BuildingAI 使用自定义 JWT + DB `user_token` 表进行认证，通过 `Authorization: Bearer <token>` 传递
- 无内置 OIDC/SAML/LDAP 支持，需要通过 BFF 做桥接
- BFF 用 Keycloak 验证用户后，根据 email/username 在 BuildingAI 中 JIT 创建或查找用户
- BFF 调用 BuildingAI 内部登录逻辑生成 JWT，并写入 `user_token` 表
- Keycloak 的 role 映射到 BuildingAI 的单一 role（默认角色）
- 具体应用、知识库、模型、MCP 工具的访问权限由 BuildingAI 管理员在系统内部配置
- BFF 需要处理 BuildingAI token 过期和刷新（通过 `x-new-token` 响应头）

### 3.6 与现有代码的衔接

当前仓库已实现：
- `gateway/`：Bun + Hono 的 OpenCode 多用户网关，端口 `9090`
- `keycloak/`：Docker Compose + realm import，端口 `9091`，realm `opencode`

演进策略：
1. **简化 OpenCode 进程模型**：从当前每用户动态端口，改为单实例 OpenCode 运行在 `:4096`；设置 `OPENCODE_SERVER_PASSWORD` 仅允许 BFF 访问
2. **保留现有 gateway 核心**：auth、session、proxy 逻辑
3. **扩展为 tri-system gateway**：新增 `dataease.ts` 和 `buildingai.ts` 模块，负责 X-DE-TOKEN 换取和 BuildingAI JWT 生成
4. **新增平台数据库**：引入 MySQL，存储 token 映射、审计日志、流水线、API key、通知配置等
5. **迁移 realm**：将 `opencode` realm 改名为 `aiplatform`，增加 DataEase/BuildingAI 所需的角色和 group
6. **新增前端**：从 Hono 模板渲染过渡到 React/Vue SPA，gateway 提供 `/api/*` 和 `/auth/*`
7. **统一端口**：最终统一为 `3000`（前端）/`3001`（gateway）/`8080`（Keycloak）/`:4096`（OpenCode）

### 3.7 平台数据模型

平台数据库采用 **MySQL 8.0**，通过 Drizzle ORM（与 OpenCode 技术栈一致）或 TypeORM 访问。需要持久化的核心数据：

> 当前代码中 gateway 使用 `config.yaml` 管理用户、项目、OIDC 配置。演进时用户/项目仍由 Keycloak 管理，但平台级数据（token 映射、审计日志、流水线、子系统连接、API key、通知配置等）迁移到 MySQL。

| 实体 | 存储 | 说明 |

### 3.8 缓存与性能策略

1. **Token 缓存**：
   - DataEase X-DE-TOKEN 和 BuildingAI JWT 缓存到 Redis，避免每次请求都重新换取
   - 缓存 key：`token:{userId}:{system}`
   - 过期时间略短于 token 实际有效期，BFF 自动刷新

2. **健康检查缓存**：
   - 后端服务状态每 30 秒探测一次，结果缓存 30 秒
   - 避免前端轮询直接打到下游系统

3. **统计聚合缓存**：
   - 工作台统计卡片数据缓存 5 分钟
   - 实时性要求不高的数据可适当延长

### 3.9 安全设计

1. **OpenCode 无认证的安全控制**：
   - 生产环境必须设置 `OPENCODE_SERVER_PASSWORD` 或仅允许 BFF 内网访问
   - BFF 对 `/api/code/*` 做平台身份校验，避免未授权访问

2. **Token 安全**：
   - DataEase X-DE-TOKEN、BuildingAI JWT 只存储在服务端 Redis，不返回给前端
   - 前端只持有平台 Keycloak JWT

3. **SSO 安全**：
   - Keycloak 使用 PKCE 流程
   - redirect_uri 严格白名单
   - 定期轮换 Keycloak client secret

4. **审计与告警**：
   - BFF 记录所有 API 调用
   - 异常访问（如非开发者访问代码工场）触发审计告警

### 3.10 错误处理与降级策略

| 场景 | 处理策略 |
|------|----------|
| Keycloak 不可用 | 已登录用户继续使用本地 session 缓存；新用户无法登录；提示维护状态 |
| DataEase 不可用 | 数据洞察模块显示"服务不可用"；不影响代码工场和 AI大脑 |
| BuildingAI 不可用 | AI大脑模块显示"服务不可用"；不影响其他模块 |
| OpenCode 不可用 | 代码工场显示"服务不可用"；BFF 不再代理到该实例 |
| DataEase token 换取失败 | 重试 3 次；失败后降级为只读或引导用户联系管理员 |
| BuildingAI token 生成失败 | 重试 3 次；失败后降级为只读 |
| 智能流水线执行失败 | 记录错误日志；通知触发者；支持手动重试 |

## 六、总结

**企业AI智造平台** 通过路线 A（API 网关模式），在不修改三个开源后端的前提下，用统一前端 + BFF 网关层将它们整合为一个有机整体。核心创新在于**智能流水线**——让数据在三个系统间自动流转，实现"数据→洞察→决策→行动"的完整闭环。

三个系统的角色分工：
- **OpenCode（代码工场）**：开发者的 AI 助手，负责"创造"能力。它是一个 agentic AI coding agent，平台菜单聚焦会话、审查、Diff、项目初始化、GitHub 自动化与配置管理。
- **DataEase（数据洞察）**：业务人员的数据之眼，负责"看见"能力。它是一个 BI 平台，核心能力是仪表板、图表、数据集、数据源和报表。
- **BuildingAI（AI大脑）**：全员的智能引擎，负责"思考"能力。它是一个 AI agent 平台，核心能力是智能体、知识库、模型和 MCP 工具。

当前代码已实现 OpenCode 多用户网关 + Keycloak，是平台的坚实基础。下一步将扩展为 tri-system BFF、统一前端和智能流水线，最终交付完整的企业 AI 智造平台。
